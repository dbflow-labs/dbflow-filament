# DBFlow Filament v1.1 Runtime Visibility (Stage 1.1-E)

**Status:** Accepted

## Standard Package Responsibilities

The Standard Filament package (`dbflowlabs/filament`) provides read-only operational visibility for Core v1.1 runtime features:

- Assignment provenance (original vs effective actor, assignment source)
- Delegation lifecycle visibility (list only)
- SLA event state on workflow instances
- Reliable Action execution and attempt metadata
- Redacted webhook execution metadata
- Extended workflow timeline mappings

Standard continues to own existing approval operations only:

- Approve
- Reject
- Reassign (when enabled)

## Pro Package Responsibilities

`dbflowlabs/filament-pro` owns management and configuration surfaces:

- Visual workflow builder / canvas
- Delegation create, revoke, migrate
- SLA policy configuration
- Action retry, skip, cancel, recovery workbench
- Premium operational dashboards

Standard must not expose disabled Pro actions or configuration forms.

## Authorization Rules

| Ability | Default config key | Purpose |
|---------|-------------------|---------|
| `delegations.view_any` | `dbflow.delegations.view_any` | Delegation list page |
| `sla_events.view` | `dbflow.sla_events.view` | SLA sections on instance detail |
| `action_executions.view_any` | `dbflow.action_executions.view_any` | Action execution list + instance summary |
| `action_executions.view` | `dbflow.action_executions.view` | Action execution detail page |
| `action_attempts.view` | `dbflow.action_attempts.view` | Attempt history on execution detail |
| `webhook_metadata.view` | `dbflow.webhook_metadata.view` | Webhook destination/idempotency columns |

Operational pages return `canAccess() === false` when permission is denied. There is no permissive fallback for sensitive operational data.

Task inbox access (`dbflow.tasks.view`) does not imply global execution visibility.

## Display Contracts

- **Assignment provenance:** `WorkflowTaskAssignment::assignmentSourceOrDirect()`, `originalAssigneeUserId()`, `effectiveAssigneeUserId()`
- **Delegation lifecycle:** `WorkflowDelegation::lifecycle()` (Core-owned rules)
- **Overdue state:** `WorkflowTask::overdue_at` (never UI-only recalculation)
- **Action status:** `WorkflowActionExecution` ledger fields
- **Webhook metadata:** redacted summaries via `SafeMetadataPresenter`

Forbidden in Standard UI:

- Raw snapshots (`payload_snapshot`, `node_snapshot`, `policy_snapshot`)
- Secret values, Authorization headers, cookies, signing material
- Retry / skip / cancel / delegation management actions

## Query Strategy

- My Tasks: existing `WorkflowTaskQueryService` pending query with indexed filters on `assignment_source`, `workflowTask.due_at`, `workflowTask.overdue_at`
- Instance detail: bounded queries (`runtime_detail_limit`, default 100) for SLA events and action executions
- List pages: paginated Filament tables with `with()` / `withCount()` only; no full snapshot loading

## Timeline Mappings

`WorkflowInstanceTimelinePresenter` maps all Core `WorkflowLogEvent` values with translation fallback to `WorkflowLogEvent::label()` and finally `timeline.unknown`.

## Backward Compatibility

- v1.0 assignments without provenance columns fall back to `assignee_user_id` via Core model helpers
- Existing approve/reject/reassign actions unchanged
- Core `^1.0` consumers must remain on Filament `^1.0`; Filament `^1.1` requires Core `^1.1`

## Non-Goals (Stage 1.1-E)

- Pro builder or management UI
- Action recovery workbench
- SLA/delegation configuration
- Raw request/response viewers
